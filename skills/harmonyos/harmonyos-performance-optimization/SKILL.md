# HarmonyOS / OpenHarmony 应用性能优化

## Overview

本 Skill 用于指导 HarmonyOS / OpenHarmony 应用的性能分析、方案设计、代码评审与优化落地。核心参考 OpenHarmony 6.0 Release 文档《应用开发性能优化入门引导》，围绕“四要素”建立可测试、可验证、可复用的优化流程：

1. 合理使用并行化、预加载和缓存；
2. 尽量减少布局嵌套层数；
3. 合理管理状态变量；
4. 合理使用系统接口，避免冗余操作。

性能优化必须先定位瓶颈，再制定方案，最后通过指标验证，避免仅凭主观体验修改代码。

## When to Use

- 页面启动慢、首屏渲染慢、Tab 切换慢、点击响应慢。
- 列表滚动卡顿、动画掉帧、页面布局复杂导致渲染压力大。
- ArkUI 状态变量更新范围过大，导致组件频繁刷新。
- 生命周期、回调、日志、Trace 或系统接口使用不当，造成主线程阻塞或额外开销。
- 需要为 HarmonyOS 性能问题编写需求分析、优化方案、测试方案或验收标准。

Don't use for:

- 与性能无关的 UI 视觉调整。
- 未定义指标、无法复现、无法观测的“感觉变慢”问题；此类问题应先补充复现场景和性能指标。

## Optimization Principles

| 原则 | 要求 | 验证方式 |
|---|---|---|
| 先测试 | 优化前先记录基线数据 | Trace、Profiler、日志统计、录屏对比 |
| 小步改动 | 每次只解决一个明确瓶颈 | 单项指标前后对比 |
| 主线程优先 | UI 线程避免耗时任务 | 检查生命周期、事件回调、build 路径 |
| 精准刷新 | 状态变化只影响必要组件 | 检查 @State/@Prop/@Link/@Observed 使用范围 |
| 可回归 | 优化结果可重复验证 | 固定设备、版本、账号、数据量、操作路径 |

## Performance Analysis Workflow

### 1. 明确问题与指标

先把“慢”拆成可观测指标：

| 问题类型 | 建议指标 |
|---|---|
| 启动慢 | 冷启动耗时、首帧时间、首页可交互时间 |
| 响应慢 | 点击到 UI 更新耗时、接口返回到渲染完成耗时 |
| 卡顿掉帧 | FPS、长帧数量、滑动丢帧率 |
| 内存压力 | 页面进入/退出内存变化、对象释放情况 |
| CPU 压力 | 主线程耗时、后台任务耗时、峰值 CPU |

### 2. 建立基线

- 固定设备型号、系统版本、应用版本、网络环境和数据量。
- 至少记录优化前后的关键路径耗时。
- 对用户可感知场景，保留复现步骤和必要截图/录屏。

### 3. 定位瓶颈

按优先级排查：

1. 生命周期中是否执行耗时任务；
2. build 或渲染路径是否存在大量计算、同步 IO、复杂对象创建；
3. 布局是否嵌套过深；
4. 状态变量更新是否导致过多组件刷新；
5. 高频回调、日志、Trace、监听注册是否冗余；
6. 图片、Web、网络数据是否缺少预加载或缓存。

### 4. 设计优化方案

每个优化项必须包含：

- 问题现象；
- 根因判断；
- 修改范围；
- 风险点；
- 测试用例；
- 验收指标。

## Element 1: 并行化、预加载与缓存

### 适用场景

- `aboutToAppear`、页面初始化、Tab 首次切换中存在网络请求、数据解析、图片处理等耗时任务。
- 首页可见内容依赖过多同步加载。
- Web 页面、图片资源或列表数据加载影响首屏展示。

### 实施建议

| 优化方向 | 做法 | 注意事项 |
|---|---|---|
| 多线程 | 使用 TaskPool 等能力处理 CPU 密集或耗时任务 | 不要在 UI 生命周期阻塞等待结果 |
| 异步并发 | 多个独立请求并发执行 | 控制失败处理和取消逻辑 |
| 预加载 | 对 Tab、下一页、Web、图片、列表数据提前加载 | 避免预加载过度造成内存和网络浪费 |
| 缓存 | 缓存稳定数据、图片、计算结果 | 明确失效策略，避免脏数据 |
| 延后处理 | 首屏非必要任务延后到首帧之后 | 不影响核心交互和业务正确性 |

### 代码审查重点

- `aboutToAppear` 内只做轻量初始化，不放同步耗时任务。
- 点击、滑动、页面切换回调中不执行大循环、同步 IO 或复杂计算。
- TaskPool/Worker 任务结果回到 UI 后，只更新必要状态。
- 缓存有容量、过期、清理策略。

## Element 2: 减少布局嵌套层数

### 适用场景

- 页面由大量 `Column` / `Row` / `Stack` 层层嵌套组成。
- 列表项结构复杂，滑动时出现卡顿。
- UI 层级过深导致测量、布局、绘制成本上升。

### 实施建议

| 优化方向 | 做法 |
|---|---|
| 扁平化布局 | 移除无意义容器，合并相邻布局层 |
| 相对/绝对定位 | 对固定结构使用相对布局或绝对定位减少层级 |
| Grid/GridRow | 对网格、复杂列表使用更匹配的布局组件 |
| 自定义布局 | 对高频复杂布局封装自定义布局逻辑 |
| 列表项精简 | 列表 item 中避免深层嵌套和重复装饰组件 |

### 验收建议

- 统计关键页面或列表项布局层级是否下降。
- 对滚动列表验证 FPS、长帧、掉帧率是否改善。
- 确认不同屏幕尺寸、字体大小、横竖屏下布局未破坏。

## Element 3: 合理管理状态变量

### 适用场景

- 一个状态变量变化导致整个页面或大量组件刷新。
- 对象级状态变量成员频繁变化，关联组件过多。
- 列表中每个 item 共用大对象状态，局部变化触发全量刷新。

### 实施建议

| 优化方向 | 做法 |
|---|---|
| 缩小状态作用域 | 状态放到真正需要刷新的最小组件内 |
| 拆分状态 | 将大对象拆成更细粒度字段或局部状态 |
| 减少关联组件 | 避免一个状态变量被过多组件读取 |
| 控制对象成员更新 | 对对象级状态变量，减少无关成员变化引发刷新 |
| 单向数据流 | 明确状态来源，减少重复状态和派生状态不一致 |

### 代码审查重点

- `@State` 是否定义在过高层级。
- 是否存在“为了方便”把整页数据对象传给多个子组件。
- 列表 item 是否能独立更新，而不是触发整页刷新。
- 派生数据是否可以缓存或按需计算。

## Element 4: 合理使用系统接口，避免冗余操作

### 适用场景

- 高频回调内执行日志、Trace、状态更新或系统接口调用。
- 页面多次进入后重复注册监听，退出时未注销。
- Debug 日志、Trace 标记在生产路径中保留过多。

### 实施建议

| 优化方向 | 做法 |
|---|---|
| 高频回调节流 | 对滚动、输入、传感器、动画等回调进行节流/防抖 |
| 删除冗余日志 | 生产环境移除高频日志和无效 Trace |
| 成对注册/注销 | 页面进入注册监听，页面退出及时注销 |
| 避免重复调用 | 缓存系统接口返回的稳定结果 |
| 控制状态更新 | 回调中只在值真正变化时更新状态 |

## Test-First Requirement

优化前必须先设计测试。推荐测试结构：

| 阶段 | 内容 | 产物 |
|---|---|---|
| RED | 复现性能问题，记录基线 | 复现步骤、基线指标 |
| GREEN | 实施最小优化 | 修改记录、测试结果 |
| REFACTOR | 清理重复逻辑、降低耦合 | 代码审查记录 |
| VERIFY | 对比优化前后指标 | 验收报告 |

### 测试用例模板

```markdown
## 性能测试用例：<场景名称>

### 前置条件
- 设备：
- 系统版本：
- 应用版本：
- 网络/数据量：

### 操作步骤
1. 启动应用 / 进入页面 / 执行动作。
2. 重复操作 3-5 次，排除偶发波动。
3. 记录 Trace / Profiler / 日志指标。

### 基线指标
| 指标 | 优化前 | 优化后 | 目标 |
|---|---:|---:|---:|
| 首屏耗时 |  |  |  |
| 点击响应 |  |  |  |
| 长帧数量 |  |  |  |

### 验收标准
- 关键指标达到目标值；
- 功能行为与优化前一致；
- 无新增崩溃、白屏、数据错乱、状态不同步问题。
```

## Common Pitfalls

1. **未建立基线就优化。** 没有优化前数据，无法证明修改有效。
2. **把所有任务都丢到后台。** 后台任务过多会增加调度和内存压力，应只迁移真正耗时且可并发的任务。
3. **预加载无边界。** 预加载必须限制数据量、生命周期和取消策略。
4. **缓存无失效策略。** 缓存可能带来脏数据、内存增长和状态不一致。
5. **只减少布局层级但破坏适配。** 扁平化后必须验证多屏幕、多语言、字体缩放。
6. **状态拆分过度。** 状态太碎会提升维护成本，应以减少无效刷新为目标，而不是机械拆分。
7. **高频日志未清理。** 高频日志会显著影响性能，尤其是滚动、动画、输入场景。
8. **注册监听不注销。** 会造成重复回调、内存泄漏和页面返回后仍执行逻辑。

## Verification Checklist

- [ ] 已明确性能问题类型和量化指标。
- [ ] 已记录优化前基线。
- [ ] 已确认耗时任务未阻塞 UI 主线程。
- [ ] 已检查关键页面布局层级，移除无意义嵌套。
- [ ] 已检查状态变量作用域和刷新范围。
- [ ] 已检查高频回调、日志、Trace、监听注册。
- [ ] 已完成优化前后对比测试。
- [ ] 已完成核心功能回归测试。
- [ ] 已记录风险、限制和后续优化项。
